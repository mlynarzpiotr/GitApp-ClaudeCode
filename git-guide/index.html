Co to jest Git?

Git to system kontroli wersji. Wyobraz sobie, ze masz przycisk "Cofnij" dla calego projektu - to wlasnie robi Git. Zapisuje kazda zmiane, ktorej dokonasz, i pozwala wrocic do dowolnego momentu w historii.

Analogia

Pomysl o Git jak o grze komputerowej z save'ami. Robisz "save" (commit) po kazdym waznym kroku. Jesli cos popssujesz, mozesz zaladowac poprzedni save.

Dlaczego warto?

  • Nigdy nie stracisz swojej pracy
  • Mozesz eksperymentowac bez ryzyka
  • Widzisz kto, co i kiedy zmienil
  • Mozesz pracowac z innymi nad tym samym kodem
Git vs GitHub - jaka roznica?

Git = narzedzie na Twoim komputerze (dziala offline)

GitHub = serwis internetowy, ktory przechowuje Twoje repozytoria w chmurze i umozliwia wspolprace

Instalacja i konfiguracja

Zanim zaczniesz, musisz zainstalowac Git i ustawic swoje dane.

macOS

▼

Otwierasz Terminal i wpisujesz:

Terminal 
$ xcode-select --install

Git jest juz wbudowany w macOS - ta komenda zainstaluje narzedzia deweloperskie razem z Git.

Windows

▼

Pobierz instalator ze strony git-scm.com i zainstaluj z domyslnymi ustawieniami. Dostaniesz Git Bash - specjalny terminal do pracy z Git.

Linux

▼
Terminal 
$ sudo apt install git    # Ubuntu/Debian
$ sudo dnf install git    # Fedora

Pierwsza konfiguracja

Po instalacji powiedz Gitowi kim jestes (te dane beda widoczne przy kazdym uczestniku zmian):

Konfiguracja - wykonaj raz 
# Ustaw swoje imie i nazwisko
$ git config --global user.name "Twoje Imie"

# Ustaw swoj email (ten sam co na GitHubie)
$ git config --global user.email "twoj@email.com"

# Sprawdz czy wszystko sie zapisalo
$ git config --list
3

Podstawowe komendy

Oto komendy, ktorych bedziesz uzywac codziennie.

Tworzenie repozytorium

Nowy projekt 
# Wejdz do folderu projektu
$ cd moj-projekt

# Zainicjuj Git w tym folderze
$ git init
# Odpowiedz: Initialized empty Git repository in .../moj-projekt/.git/
Co robi git init?

Tworzy ukryty folder .git w Twoim projekcie. Tam Git przechowuje cala historie zmian. Nie usuwaj tego folderu!

Sprawdzanie stanu

Najczesciej uzywana komenda 
$ git status

# Pokaze Ci:
# - na jakiej jestes galezi (branch)
# - jakie pliki zostaly zmienione
# - jakie pliki sa gotowe do commita
# - jakie pliki sa nowe i niesledzone

Zapisywanie zmian (commit)

Dodawanie i commitowanie 
# Krok 1: Dodaj pliki do "poczekalni" (staging area)
$ git add nazwa-pliku.txt      # dodaj jeden plik
$ git add .                     # dodaj WSZYSTKIE zmienione pliki

# Krok 2: Zapisz zmiany z opisem co zrobiles
$ git commit -m "Dodano strone glowna"
Zasada dobrych commitow

Pisz wiadomosci commitow po angielsku lub polsku, ale zawsze w trybie rozkazujacym: "Dodaj formularz logowania" zamiast "Dodalem formularz logowania". Commituj czesto - male, logiczne porcje zmian.

Przegladanie historii

Historia commitow 
# Pelna historia
$ git log

# Skrocona historia (jeden commit = jedna linia)
$ git log --oneline

# Ladny graf z branchami
$ git log --oneline --graph --all

Sprawdzanie roznic

Co sie zmienilo? 
# Pokaz niezapisane zmiany
$ git diff

# Pokaz zmiany gotowe do commita (w staging area)
$ git diff --staged

Jak dziala Git - workflow

Git ma 3 glowne "miejsca" w ktorych zyja Twoje pliki:

Working Directory
Twoje pliki
→
Staging Area
Poczekalnia
→
Repository
Historia
git add
git commit
1

Working Directory (katalog roboczy)

To Twoje normalne pliki na dysku. Edytujesz je jak zwykle.

2

Staging Area (poczekalnia)

Uzywasz git add zeby wybrac ktore zmiany chcesz zapisac. To jak wkladanie rzeczy do paczki przed wyslaniem.

3

Repository (repozytorium)

Uzywasz git commit zeby trwale zapisac zmiany w historii. Jak wyslanie paczki - nie da sie juz jej zmienic.

5

Branche (galerie)

Branche pozwalaja pracowac nad roznymi funkcjami jednoczesnie, bez mieszania kodu. To jak rownolegle wymiary w grze - mozesz eksperymentowac w jednym, a drugi zostaje nietkniety.

main:      o---o---o---o---o
                \       /
feature:         o---o---o
Praca z branchami 
# Zobacz wszystkie branche
$ git branch

# Stworz nowy branch
$ git branch nazwa-brancha

# Przejdz na inny branch
$ git checkout nazwa-brancha

# Stworz nowy branch I od razu na niego przejdz (skrot!)
$ git checkout -b nowy-branch

# Polacz branch z main (merge)
$ git checkout main           # wroc na main
$ git merge nazwa-brancha     # polacz zmiany

# Usun branch (po merge)
$ git branch -d nazwa-brancha
Konwencja nazw branchy

Uzywaj opisowych nazw: feature/login-form, fix/navbar-bug, docs/readme-update

6

GitHub - praca z chmura

GitHub przechowuje Twoj kod w chmurze. Dzieki temu masz backup, mozesz wspolpracowac z innymi i pokazac swoje projekty swiatu.

Pierwsze kroki z GitHub

1

Zaloz konto na github.com

Uzyj tego samego emaila co w konfiguracji Git.

2

Stworz nowe repozytorium

Kliknij "New" na GitHub. Daj nazwe projektowi. Nie zaznaczaj "Initialize with README" jesli juz masz lokalny projekt.

3

Polacz lokalny projekt z GitHub

Wykonaj komendy ponizej.

Polaczenie z GitHub 
# Dodaj adres zdalnego repozytorium
$ git remote add origin https://github.com/TWOJ-LOGIN/nazwa-repo.git

# Wyslij kod na GitHub (pierwszy raz)
$ git push -u origin main

# Kolejne razy wystarczy:
$ git push

Klonowanie istniejacego projektu

Kopiowanie projektu z GitHub 
# Pobierz projekt z GitHub na swoj komputer
$ git clone https://github.com/uzytkownik/nazwa-repo.git

# Wejdz do pobranego folderu
$ cd nazwa-repo

Synchronizacja

Push i Pull 
# Wyslij swoje zmiany na GitHub
$ git push

# Pobierz zmiany innych z GitHub
$ git pull

# Sprawdz co sie zmienilo na GitHub (bez pobierania)
$ git fetch

Pull Requesty i wspolpraca

Pull Request (PR) to prosba o dolaczenie Twoich zmian do glownego kodu. To serce wspolpracy na GitHubie.

1

Stworz branch z nowa funkcja

Terminal 
$ git checkout -b feature/nowa-funkcja
2

Wprowadz zmiany i commituj

Terminal 
$ git add .
$ git commit -m "Dodaj nowa funkcje"
3

Wyslij branch na GitHub

Terminal 
$ git push -u origin feature/nowa-funkcja
4

Stworz Pull Request na GitHubie

Wejdz na GitHub → zobaczysz przycisk "Compare & pull request" → dodaj opis → kliknij "Create pull request"

5

Code Review i Merge

Inni przegladaja Twoj kod, komentuja, sugeruja zmiany. Gdy wszystko jest OK - klikasz "Merge pull request".

Plik .gitignore

Stworz plik .gitignore w glownym folderze projektu. Wpisz w nim nazwy plikow i folderow, ktore Git ma ignorowac:

.gitignore 
node_modules/
.env
.DS_Store
dist/
*.log

Najczestsze problemy i rozwiazania

Chce cofnac ostatni commit (ale zachowac zmiany)

▼
Terminal 
$ git reset --soft HEAD~1

Twoje pliki zostana nienaruszone, ale commit zostanie cofniety.

Zmienialem pliki, ale chce wrocic do ostatniego commita

▼
Terminal 
# Cofnij zmiany w jednym pliku
$ git checkout -- nazwa-pliku.txt

# Cofnij WSZYSTKIE zmiany (UWAGA: stracisz niezapisane zmiany!)
$ git checkout -- .
Ostrzezenie

Ta komenda trwale usunie niezacommitowane zmiany. Upewnij sie, ze tego chcesz!

Mam konflikt przy merge - co robic?

▼

Konflikty powstaja gdy dwie osoby edytowaly ten sam fragment kodu. Git oznaczy konflikty w pliku:

Plik z konfliktem 
<<<<<<< HEAD
Twoja wersja kodu
=======
Wersja kodu z drugiego brancha
>>>>>>> nazwa-brancha

Rozwiazanie:

  1. Otworz plik i wybierz ktora wersje chcesz zachowac
  2. Usun znaczniki (<<<<, ====, >>>>)
  3. Zapisz plik
  4. git add . i git commit

Przypadkowo dodalam plik do staging area

▼
Terminal 
# Usun plik ze staging area (plik zostanie na dysku)
$ git restore --staged nazwa-pliku.txt

Chce zmienic wiadomosc ostatniego commita

▼
Terminal 
$ git commit --amend -m "Nowa wiadomosc"
Uwaga

Jesli juz wyslales commit na GitHub (git push), nie zmieniaj go! Moze to skomplikowac sprawy dla reszty zespolu.

Chce zobaczyc co zmienil konkretny commit

▼
Terminal 
# Pokaz zmiany z konkretnego commita
$ git show abc1234

# (abc1234 to poczatek hash commita - znajdziesz go w git log)

Funkcje i cechy GitHuba

GitHub to znacznie wiecej niz tylko miejsce na kod. Oto najwazniejsze funkcje, ktore powinienes znac:

📁

Repozytoria (Repos)

Centralne miejsce Twojego projektu. Kazde repo zawiera kod, historie zmian, dokumentacje i ustawienia. Moze byc publiczne (widoczne dla wszystkich) lub prywatne (tylko dla Ciebie i zaproszonych osob).

Podstawa
⚠️

Issues (Zgloszenia)

System zarzadzania zadaniami wbudowany w GitHub. Mozesz zglaszac bledy, proponowac nowe funkcje, prowadzic dyskusje. Kazdy issue ma numer, etykiety (labels) i moze byc przypisany do osoby.

Wspolpraca
🔀

Pull Requesty (PR)

Serce wspolpracy. Proponujesz zmiany, inni je przegladaja (code review), komentuja i zatwierdzaja. PR laczy sie z branchem i moze byc automatycznie testowany przed merge.

Wspolpraca
⚡

GitHub Actions (CI/CD)

Automatyzacja zadan. Mozesz ustawic aby po kazdym pushu automatycznie uruchamialy sie testy, budowal sie projekt, a nawet wdrazal na serwer. Konfigurujesz to plikami YAML w folderze .github/workflows/.

Zaawansowane
🌐

GitHub Pages

Darmowy hosting stron internetowych prosto z repozytorium. Idealne dla portfolio, dokumentacji lub prostych stron. Twoja strona bedzie dostepna pod adresem uzytkownik.github.io/repo.

Darmowe
👥

Organizacje i zespoly

Grupuj repozytoria i ludzi w organizacje. Przydzielaj role i uprawnienia, zarzadzaj zespolami. Idealne dla firm i projektow open-source z wieloma uczestnikami.

Wspolpraca
📚

README i dokumentacja

Plik README.md to wizytowka Twojego projektu - wyswietla sie na stronie glownej repo. Pisz go w Markdown. Dodaj opis projektu, instrukcje instalacji i przyklad uzycia.

Podstawa
🔌

Forki (Rozgalezienia)

Skopiuj cudze repozytorium na swoje konto. Mozesz eksperymentowac z kodem bez wplywu na oryginal. Gdy masz poprawke - wysylasz Pull Request do oryginalnego projektu.

Wspolpraca
⭐

Stars i Explore

Daj "gwiazdke" projektom, ktore Ci sie podobaja - to jak zakladka i jednoczesnie wsparcie dla autora. Zakladka Explore pomaga odkrywac popularne i nowe projekty.

Darmowe
🔒

Bezpieczenstwo

GitHub automatycznie skanuje Twoj kod pod katem znanych luk (Dependabot). Powiadamia Cie o podatnosciach w zaleznosciach i moze automatycznie tworzyc PR z poprawkami.

Zaawansowane

Kluczowe elementy interfejsu GitHub

Zakladki w repozytorium

  • Code - przegladanie plikow i katalogow projektu
  • Issues - lista zgloszen, bledow i zadan do zrobienia
  • Pull requests - lista oczekujacych zmian do przegladniecia
  • Actions - automatyczne procesy (testy, budowanie, wdrazanie)
  • Projects - tablice Kanban do zarzadzania zadaniami (jak Trello)
  • Wiki - miejsce na rozbudowana dokumentacje projektu
  • Settings - ustawienia repo (uprawnienia, branche, webhooki)

Twoj profil GitHub

  • Contribution graph - zielone kratki pokazujace Twoja aktywnosc kazdego dnia
  • Pinned repos - przypnij najlepsze projekty na gorze profilu
  • Profile README - stworz repo o nazwie swojego loginu i dodaj README.md - wyswietli sie na profilu
  • Followers/Following - obserwuj innych deweloperow i ich projekty
GitHub to Twoje portfolio

Rekruterzy i firmy czesto przegladaja profile GitHub kandydatow. Zadbaj o czytelne README w projektach, regularne commity i porzadna historie zmian. To najlepsza wizytowka programisty!

Darmowe vs platne

Darmowe konto daje Ci: nieograniczone publiczne i prywatne repozytoria, GitHub Pages, Actions (2000 minut/miesiac), Issues, Pull Requesty i Codespaces (60h/miesiac).

Platne plany (Pro/Team/Enterprise) dodaja: wiecej minut Actions, zaawansowane bezpieczenstwo, wymagane code reviews, CODEOWNERS i wiecej.

Sciagawka - wszystkie komendy

Wszystkie najwazniejsze komendy w jednym miejscu. Wracaj tutaj gdy potrzebujesz!

Konfiguracja

git config --global user.name "..." Ustaw imie
git config --global user.email "..." Ustaw email
git config --list Pokaz config

Tworzenie projektu

git init Nowe repo
git clone <url> Klonuj repo

Codzienne komendy

git status Stan plikow
git add <plik> Dodaj do stage
git add . Dodaj wszystko
git commit -m "..." Zapisz zmiany
git diff Pokaz roznice

Branche

git branch Lista branchy
git checkout -b <nazwa> Nowy branch
git checkout <nazwa> Zmien branch

GitHub (zdalne repo)

git remote add origin <url> Dodaj remote
git push Wyslij zmiany
git pull Pobierz zmiany
git fetch Sprawdz zmiany
git push -u origin <branch> Push nowy branch

Cofanie zmian

git restore --staged <plik> Cofnij z stage
git checkout -- <plik> Cofnij edycje
git reset --soft HEAD~1 Cofnij commit
git commit --amend -m "..." Zmien wiadomosc
